Abstract

Selectors API defines four methods for retrieving Element nodes in the DOM using a group of selectors as defined in [Selectors].

Introduction

This section is non-normative.

It is often desirable to perform script and or DOM operations on a specific set of elements in a document. Selectors, mostly used in CSS context, provides a way of matching such a set of elements. This specification introduces two methods which take a group of selectors (often simply referred to as selector) as argument and return the matched elements as result. [Selectors] [CSS21]

With these methods it's easier to match a set of Element nodes based on certain criteria. So instead of having to filter the result of a getElementsByTagName() call authors can directly "filter" in the query.

Examples

This section is non-normative.

Some [ECMAScript] examples:

<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en">
  <head> … </head>
  <body>
    <p>
      I like graphics:
      <svg xmlns="http://www.w3.org/2000/svg" …> … </svg>
      <svg xmlns="http://www.w3.org/2000/svg" …> … </svg>
    </p>
    <p id="x" xmlns:svg="http://www.w3.org/2000/svg">
      They are great:
      <svg:svg …> … </svg:svg>
    </p>
  </body>
</html>

If you apply the following script to the above XML document … 

function resolver(str) {
  var prefixes = {
    h:  "http://www.w3.org/1999/xhtml",
    g: "http://www.w3.org/2000/svg"}
  return prefixes[str];
}
var a = document.get("#x");
var b = document.getAll("h|p > g|svg", resolver);
var c = a.get("svg, span");

… a will be the first element in the document with an ID of x. b will be a StaticNodeList containing all the nodes svg elements in the http://www.w3.org/2000/svg namespace whose parent is the p element in the http://www.w3.org/2000/svg namespace. c will be a svg or span element (whichever comes first) which is a descendent of the first element in the document with an ID of x. So in this case it will be the svg element.

function test(){
  var d = document.getAll(":root"),
      e = document.documentElement;
  return d[0] === e;
}

test() would return true when both getAll() and :root are supported. Note that this example is just to illustrate how it would work. In the real world you would use get() for things that can only yield a single result and for selecting the root element you would simply use document.documentElement.

var f = document.get("p.warning, p.alert");

The above code snippet shows that besides using a single selector you can also pass a group of selectors (basically comma-separated selectors) as argument. f would contain a StaticNodeList with all p elements in the document that have a warning or alert class.

var g = document.get("#foo, #bar");

g would contain the first element in the document either having an ID of foo or bar (or both).

Terminology and Conventions

The terminology used in this specification is that from [Selectors].

The algorithms described in this specification are there for clarity. User agents are not required to follow them as long as the result is the same. In doing so user agents may assume that the object implementing the NSResolver interface (or ECMAScript Function) only relies on scoped variables, doesn't invoke processes outside the object and returns consistent results when its lookupNamespaceURI method is invoked.

Interoperability Considerations

User agents are allowed to optimize the algorithms described within this specification. This entails that some might use the object implementing the NSResolver interface more than others. If the object (or ECMAScript Function) causes side effects this might lead to some small interoperability concerns. The same goes for not getting back identical results when the method is invoked with the same prefix as argument. Authors are strongly discouraged from creating content that does that.

Security Considerations

It is expected that implementing this specification introduces no new risks for users.

History theft is a potential security problem, because of the :visited pseudo-class in [Selectors]. :visited can already be used to query which links are visited using [CSS21] and because this specification did not introduce the problem and it can already be exploited by other means user agents don't have to take action upon it.

Another problem is hostile content. User agents should ensure they remain stable when facing a hostile NSResolver object. Potential hostile things such an object could do include:

  • Returning inconsistent results;
  • Hanging;
  • Changing the DOM during the operation.
var x = document.get("test|*:empty > test|p", resolver)
function resolver(prefix) {
  sideEffect()
  if("test" == prefix)
    return "http://example.org/test"
}

Some user agents may have decided not to resolve namespace prefixes in order to determine x. This causes sideEffect() not to be invoked.

The get() and getAll() methods

Objects implementing the Document interface must also implement the DocumentSelector interface. Likewise objects implementing the Element interface must also implement the ElementSelector interface. [DOM3Core]

interface DocumentSelector {
  Node            get(in DOMString selectors, in NSResolver nsresolver);
  StaticNodeList  getAll(in DOMString selectors, in NSResolver nsresolver);
};

interface ElementSelector {
  Node            get(in DOMString selectors, in NSResolver nsresolver);
  StaticNodeList  getAll(in DOMString selectors, in NSResolver nsresolver);
}

The get(selectors, nsresolver) method, when invoked, must return the first Element node in document order (using depth-first pre-order traversal) that matches the group of selectors (selectors), if any. Otherwise it must return null.

The getAll(selectors, nsresolver) method, when invoked, must return a StaticNodeList of all the Elements that match the group of selectors (selectors) in document order, if any. Otherwise it must return an empty StaticNodeList.

Both get() and getAll() take a group of selectors (selectors) as first argument and an NSResolver (nsresolver) as second. User agents must use the nsresolver argument to resolve namespace prefixes to namespaces or to get the default namespace. When the nsresolver argument is null user agents must ignore it. [Selectors]

When authors use namespace prefixes within selectors or want to set a default namespace they must create an object implementing the NSResolver interface and pass that (or a Function in case of ECMAScript) as argument for nsresolver. Otherwise, authors may set the argument to null or omit it if the language binding allows it.

If the given group of selectors (selectors) is invalid, the user agent must raise a SYNTAX_ERR exception. If the given group of selectors (selectors) uses namespace prefixes and the prefix can not be resolved (the nsresolver argument doesn't provide the namespace or has been omitted entirely) a NAMESPACE_ERR exception must be raised.

Not defining a default namespace is ok.

In languages that support optional arguments for methods, like ECMAScript, omitting the nsresolver argument in either the get() or getAll() method must return the same result as if the method was invoked with the nsresolver argument being null.

Using pseudo-elements in one of the selectors could mean that nothing is returned for that particular selector when it doesn't resolve in one or more Element nodes.

The get() method on the ElementSelector interface, when invoked, must return the first Element node in document order (using depth-first pre-order traversal) that matches the group of selectors (selectors) within the group of descendents of the element on which the method was invoked, if any. Otherwise it must return null. Other behavior must be identical to DocumentSelector get().

The getAll() method on the ElementSelector interface must return the element nodes that DocumentSelector getAll() would return, excluding any elements that are not descendents of the object implementing ElementSelector on which the method was invoked.

The NSResolver Interface

The NSResolver interface allows prefixes to be bound to a namespace URI. The interface is to be implemented by authors.

Other specifications may introduce methods that return an object implementing NSResolver in which case it would be implemented by the user agent.

interface NSResolver {
  DOMString          lookupNamespaceURI(in DOMString prefix);
};

The lookupNamespaceURI(prefix) method must return the namespace belonging to the prefix (prefix) or the default namespace if the prefix argument is null or the empty string.

To resolve namespace prefixes user agents must pass the prefix lowercased (A-Z maps to a-z) to the lookupNamespaceURI method. User agents must not invoke the method more than once for each unique (after lowercasing) prefix. The order in which prefixes are resolved is not defined by this specification. To get the default namespace user agents must invoke the lookupNamespaceURI() method with the empty string. User agents must never invoke the method with null as argument.

In ECMAScript (and languages that allow similar constructs) user agents must support NSResolver being implemented as a Function where the parameter of the function is identical to the prefix argument described above.

var i = 0;
function resolver(prefix) {
  i++
  var prefixes = {
    1:  "http://example.org/foo",
    2:  "http://example.org/bar"}
  return prefixes[i];
}
var x = document.getAll("tESt|x > test|x", resolver)
var y = document.getAll("x|x" > "y|x", resolver)

Variable x represents a NodeList containing all x elements in the http://example.org/foo namespace that are a child of an x element in the same namespace in document order.

Variable y either represents all x elements in the http://example.org/foo namespace that are child of an x element in the http://example.org/bar namespace in document order or represents all x elements in the http://example.org/bar namespace that are child of an x element in the http://example.org/foo namespace in document order depending on the particular implementation.

The StaticNodeList Interface

typedef StaticNodeList NodeList;

The StaticNodeList must be implemented identically to the NodeList interface as defined in [DOM3Core] with the exception that the interface, as the name suggests, is static and not live.
